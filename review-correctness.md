- Blocker: The release would fail today at the first check. `scripts/release.ts` runs `pnpm audit --audit-level low` before updating/committing (`scripts/release.ts:265-269`), and that command currently exits 1 with 6 high vulnerabilities from `changelogithub > changelogen > c12 > giget > tar` (lockfile shows `tar@6.2.1`). - Blocker: Interactive custom version input is not validated. The CLI `--version` path uses `resolveVersion` and `semverPattern` (`scripts/release.ts:233-243`), but the interactive Custom path returns the raw input directly (`scripts/release.ts:227-230`). Empty or non-semver input can be written to `package.json` (`scripts/release.ts:272-273`) and used for `v${nextVersion}` tags (`scripts/release.ts:245`), causing npm publish/version failures later.

- Note: The workflow dist-tag logic is correct for simple prereleases like `v0.1.0-beta.0` -> `beta` and stable tags -> `latest` (`.github/workflows/release.yaml:25-30`). It is fragile for valid but more complex prerelease identifiers: e.g. `v1.2.3-alpha.0.1` would produce `alpha.0`, and prerelease identifiers containing hyphens would use the text after the last hyphen because of `${TAG##*-}` (`.github/workflows/release.yaml:27`). If custom versions remain allowed, this can publish under an unexpected dist-tag.
- Note: Prerelease bumping assumes the second prerelease identifier is numeric (`scripts/release.ts:136-139`, `scripts/release.ts:196-199`). Valid semver such as `1.0.0-beta.foo` or multi-part prereleases will produce surprising next versions (for example `beta.NaN` or dropping later identifiers), so prerelease handling is only safe for the documented `tag.N` shape.
- Note: `git push --tags` pushes all local tags, not just the release tag (`scripts/release.ts:282-283`). This can publish unrelated local tags if present. Pushing the specific tag would be safer.
