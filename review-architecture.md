## Review

- Correct: `.github/workflows/release.yaml` is intentionally the same shape as `~/repos/remote-agent/.github/workflows/release.yaml`: tag-triggered publish with npm provenance, prerelease dist-tag derivation, then draft notes via `changelogithub` (`.github/workflows/release.yaml:3-53`; remote-agent same at lines 3-53). This is a reasonable minimal CD workflow for npm trusted publishing.
- Correct: pi-ralph does not need a real compile/bundle step for release: the package manifest points pi directly at source/resource directories (`package.json:49-61`), and extensions import TS source files directly (for example `extensions/set-ralph-loop.ts` imports `../src/ralph-loop/...`). The current `build` script is only an echo placeholder (`package.json:20-21`), and `pnpm build` confirms it only prints “No build step required for pi package”.
- Correct: The remote-agent-only checks were not blindly copied in full. pi-ralph omits `check:bundled-licenses`, `check:pack`, `smoke:pack`, and `docs/openapi.json` staging that remote-agent needs for a bundled CLI/web app (`remote-agent/scripts/release.ts:265-280`; pi-ralph `scripts/release.ts:265-277`).

- Blocker: `pnpm audit --audit-level low` currently blocks release. The release script runs audit first (`scripts/release.ts:265-267`), and the actual audit fails with 6 high `tar` advisories through `changelogithub > changelogen > c12 > giget > tar`. This was introduced by adding release-notes tooling (`package.json:28`) and must be resolved before `pnpm release` can succeed (dependency update or an explicit override like remote-agent’s `pnpm.overrides` for `tar`).
- Blocker: `package.json` is missing a `files` allowlist. With no `.npmignore` and no `files` field (`package.json:1-63`), `npm pack --dry-run --json` includes development and CI artifacts such as `.github/**`, `.vscode/settings.json`, `.zed/settings.json`, `flake.*`, `gatecheck.yaml`, `lefthook.yml`, `oxlint.config.ts`, `scripts/release.ts`, and all `*.test.ts`. For a pi package, source/resource directories are needed, but release/dev infrastructure should not be published. Remote-agent has an explicit allowlist (`remote-agent/package.json:13-16`). pi-ralph should add one suited to pi package discovery, e.g. `extensions`, `src`, `skills`, `prompts`, `themes`, plus README/LICENSE.
- Blocker: `skills/release/SKILL.md` is release-maintainer documentation placed under an auto-discovered package skill directory. The package manifest exports `./skills` to pi (`package.json:53-55`), and the dry-run tarball includes `skills/release/SKILL.md`; therefore consumers may receive “release” as a pi skill. This looks like package-surface pollution. Move release-flow docs to `docs/release.md` or README instead, unless it is intentionally a user-facing pi skill.

- Note: The `pnpm build` check in `scripts/release.ts:267` is redundant because `package.json:21` only echoes. It is harmless, but simpler design would either remove `pnpm build` from release checks or replace it with a meaningful package-shape check (`npm pack --dry-run` / custom allowlist check). For this package, publish correctness is more about tarball contents than building.
- Note: The release script is largely copied from remote-agent (interactive version bump, clean tree, signed commit/tag, push; `scripts/release.ts:34-285`). That is not necessarily wrong if signed local releases are a repo policy, but for pi-ralph the important missing guard is not more release orchestration—it is package contents validation.
- Note: README is not inaccurate about release flow because it does not document release at all (`README.md:71-77` only lists development commands). If release docs are desired, README should add a short maintainer section or link to `docs/release.md`; do not keep this under `skills/` unless intended for installed users.
